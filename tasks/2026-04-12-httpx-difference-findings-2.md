Run from inside `coverage-stats-extensive-examples/httpx/`, which uses **Python 3.9**.

## Executive Summary

**Statement counts are identical.** `_parse_with_coverage` correctly delegates to
coverage.py's `PythonParser` under the same Python 3.9 environment, so both tools
produce exactly the same executable line sets for all 23 files.

**All remaining differences are in branch/arc counts.** The `BranchWalker`'s
AST-based branch model disagrees with coverage.py's bytecode-based arc tracker in
three distinct ways.

---

## Why the Earlier "Python Version" Hypothesis Was Wrong

The previous hypothesis (Python 3.14 vs 3.9 bytecode differences) only applied
when comparing coverage-stats running under the project's `.venv` (Python 3.14)
against `cov-report.json` (generated by the httpx `.venv`, Python 3.9). When you
run the command above, both tools use the **same Python 3.9** `.venv`, so the
bytecode-level differences vanish entirely. Confirmed: running the same comparison
with the httpx `.venv`'s Python shows zero statement-count differences.

## Cause 1: `while True:` Counted as 2-Arc Branch by BranchWalker

**Files:** `_auth.py` (+4), `_client.py` (+6 net)

`BranchWalker.walk_branches` handles all `ast.While` nodes as 2-arc branches,
including `while True:`. But in Python 3.9, the compiler emits no conditional
jump for `while True:` — the loop body starts with an unconditional jump. Coverage.py
sees no branch at that line and counts **0 arcs** for it.

```
# Python 3.9 dis for: while True: x = 1; break
  2  >>  0 LOAD_CONST    1    ← body starts immediately (line 2)
         2 STORE_NAME    0
  3       4 JUMP_ABSOLUTE 8
         ...
# Line 1 (while True:) has NO bytecode — coverage.py doesn't count it as a branch
```

BranchWalker adds 2 arcs per `while True:`. Coverage.py adds 0.
`_auth.py` has 2 such loops → +4 arcs. `_client.py` has 4 such loops → expected
+8, net observed +6 (2 of the 4 interact with other effects, see below).

**Fix:** Skip `ast.While` nodes where `node.test` is a truthy `ast.Constant` in
`BranchWalker.walk_branches`.

## Cause 2: `AsyncFor` Not Handled by BranchWalker

**Files:** `_content.py` (-2), `_models.py` (-4)

`BranchWalker.walk_branches` line 73:
```python
if not isinstance(node, (ast.If, ast.While, ast.For)):
    continue
```
`ast.AsyncFor` is absent. Coverage.py tracks async for loops as branches.

`_content.py` has 1 `async for` → 2 missing arcs.
`_models.py` has 4 `async for` loops → 8 theoretical missing arcs, but -4 net
(some loops are in excluded regions and thus also skipped by coverage.py).

**Fix:** Add `ast.AsyncFor` to the `isinstance` check in `BranchWalker.walk_branches`.

---

## Cause 3: Single-Excluded-Target Branches Counted as 1-Arc Instead of 0

**Files:** `_config.py` (+2), contributes to `_client.py` and others

When a branch's true or false target is an excluded line, `_analyze_branches`
reduces `effective_arc_count` from 2 to 1 and still adds it to `arcs_total`.
Coverage.py treats such a branch as having only one real choice and counts
**0 arcs** for it (a branch with only one non-excluded path isn't a branch).

Example in `_config.py`:
```python
if verify is True:       # line 43 — true → 44 (excluded), false → 51 (non-excl)
    ...                  # line 44 — excluded (part of excluded block)
elif verify is False:    # line 51 — true → 52 (non-excl), false → 55 (excluded)
```

For line 43: `effective_arc_count = 2 - 1 = 1` (BranchWalker), `0` (coverage.py).
For line 51: same. Net: +2 arcs in coverage-stats for `_config.py`. ✓

**Fix:** In `_analyze_branches`, change the condition from `if effective_arc_count <= 0`
to skip at `< 2` non-excluded arcs (i.e., treat 1-effective-arc branches as 0-arc).

## Arc Coverage Discrepancy (Covered Count)

**Files:** `__init__.py` (-2 covered), `_status_codes.py` (-1 covered)

BranchWalker determines whether an arc was taken by checking if the **target line**
was executed:
```python
true_taken = body_count > 0  # body_count = executions of body_lineno
```
Coverage.py tracks the exact `(source_line, target_line)` pair. When a target line
is reachable via multiple paths, BranchWalker can mark an arc as "taken" when the
target was reached by a *different* arc (false positive), or miss it when the line
number doesn't map cleanly to the arc target.

This causes coverage-stats to report 2 and 1 fewer covered items than coverage.py
for these two files respectively.
