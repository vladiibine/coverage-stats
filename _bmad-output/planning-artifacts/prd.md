# Product Requirements Document - coverage-stats

**Author:** Vlad
**Date:** 2026-03-15

## Executive Summary

**coverage-stats** is a greenfield open-source Python developer tool that replaces the binary line-execution metric of traditional coverage tools with an intent-aware analytics model. Distributed as a pip package and pytest plugin, it allows developers to explicitly declare test ownership via a `@covers` decorator, then independently tracks four metrics per line — incidental executions, deliberate executions, incidental assert density, and deliberate assert density — using a custom `sys.settrace`-based profiler with no dependency on coverage.py. Output formats include an HTML report (coverage.py-style index with folder-level collapsible stats + per-file line-level view), JSON, and CSV.

**Target users:** Individual Python developers (primary) who want confidence that critical code is genuinely tested, not just incidentally executed. Secondary users — QA managers, architects, and CTOs — consume HTML report artifacts generated by CI pipelines to drive testing investment decisions and assess release risk.

**The problem:** Code coverage tools answer only "was this line executed?" — a signal widely acknowledged as weak. A line can be "covered" solely because an unrelated test passed through it, with no assertions targeting it. Developers cannot distinguish deliberate testing from incidental execution, and managers have no metric for test quality — only test quantity.

**Classification:** Developer tool (pip/PyPI + pytest plugin) · Python/general domain · Medium complexity · Greenfield

### What Makes This Special

**Core insight:** "coverage.py tells you what ran; coverage-stats tells you what was actually tested."

No existing open-source tool combines explicit test-to-code ownership declaration, incidental vs. deliberate execution counting, assert density measurement, independent profiling, and coverage.py-style HTML reporting. The key differentiators:

- **Intent-aware profiling** — the only tool that distinguishes *why* a line was executed
- **Assert density as a signal** — quantifies verification pressure per line, a metric no existing tool exposes
- **Hard failure on bad references** — `@covers` resolves targets just before the test runs; unresolvable references raise an exception and fail the test (suite continues), never silently ignored
- **Full independence** — custom profiler, custom reporter, zero coverage.py dependency
- **Incremental adoption** — delivers insight from the first annotation; no requirement to annotate the full test suite before the tool is useful

None of them claim to test it.

**Climax:** Alex adds `@covers(PricingCalculator.calculate_discount)` to three focused tests. Reruns. Deliberate coverage climbs to 41%. The per-line view reveals three lines in the discount logic that have never been touched by any deliberate test — including the exact line that caused the production bug.

**Resolution:** Alex opens a PR: "Add deliberate coverage annotations to pricing module." In the description: "Turns out our 78% coverage was mostly wishful thinking on this file. Here's what we're actually testing." The team adopts `@covers` as a norm. coverage-stats runs in CI. The metric that matters is no longer the overall percentage — it's the deliberate assert density on the modules that keep the business running.

### Journey 2: Alex — Error Path (Primary Developer, Bad Reference)

Six months later, a junior developer on Alex's team adds a `@covers` annotation during a refactor:

```python
@covers('pricing.calculator.DiscountCalculator.apply_discount')
def test_discount_applied():
    ...
```

The class was renamed to `PricingCalculator` in the refactor. The path no longer resolves.

**What happens:** The test suite runs. When pytest reaches `test_discount_applied`, coverage-stats tries to resolve the reference just before the test executes. It fails. An exception fires: `CoverageStatsResolutionError: Cannot resolve 'pricing.calculator.DiscountCalculator.apply_discount' — class 'DiscountCalculator' not found in module 'pricing.calculator'. Did you rename this class?` The test is marked as errored. The rest of the suite continues.

**Resolution:** The CI job fails with a clear error pointing to the stale annotation. The developer updates the reference in 30 seconds. No silent drift, no misleading metrics.

*Reveals requirements: lazy reference resolution, clear exception message with context, test-level failure (not suite-level), suite continuation after resolution error.*

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. The Intent-Aware Coverage Paradigm**

coverage-stats introduces a fundamentally new dimension to code coverage measurement: *test intent*. All existing coverage tools (coverage.py, Istanbul, JaCoCo, etc.) share the same underlying model — a line is either executed or not. coverage-stats breaks this by asking: *why* was this line executed? Was a test deliberately claiming ownership of it, or did it run as a side effect?

This is a paradigm shift, not an incremental improvement. The new primitives are:
- **Deliberate execution** — a test that explicitly claimed this code via `@covers` ran this line
- **Incidental execution** — the line ran but no test claimed it
- **Assert density** — how much verification pressure (measured in assertions) backs a given line or file

These primitives unlock insights no existing tool provides: the "coverage illusion" (high incidental coverage masking zero deliberate coverage), and assert density as a test quality signal.

**2. Declarative Test Ownership via `@covers`**

The `@covers` decorator creates a lightweight, machine-readable language for expressing test intent. It accepts Python objects, dotted string references, or lists of either — making it expressive enough for real-world test suites while remaining simple to adopt incrementally. References are resolved lazily just before each test runs; unresolvable references raise a `CoverageStatsResolutionError` and fail the test (suite continues).

**3. Assert Counting as a Coverage Dimension**

No existing coverage tool tracks assertions as a coverage signal. coverage-stats introduces *assert density* — the number of assertions fired by covering tests, divided by the number of lines in a file. This quantifies the verification pressure on code in a way that line-hit counts cannot. A line executed once by a test with 50 assertions is fundamentally different from a line executed once by a test with 0 assertions.

### Market Context & Competitive Landscape

The "coverage is a broken metric" critique is well-established in the testing community. The common conclusion is that coverage is a weak but not worthless signal. coverage-stats is the first tool to provide a *concrete alternative metric* rather than just critiquing the existing one.

Closest adjacent work:
- **coverage.py contexts** — supports dynamic contexts to tag which test ran which lines, but provides no deliberate/incidental distinction, no assert counting, and no intent-declaration API
- **mutation testing** (mutmut, cosmic-ray) — tests quality of tests by mutating code, but doesn't answer the deliberate/incidental question and is orders of magnitude slower
- No known tool combines: intent declaration + execution counting by intent + assert density + HTML reporting

### Validation Approach

- **Correctness validation:** The tool must be tested against itself — coverage-stats annotated with `@covers` should show high deliberate coverage on its own profiler and reporting code
- **"Coverage illusion" demo:** A reference example project demonstrating that 80% traditional coverage + 5% deliberate coverage is a real, common scenario serves as the primary validation of the concept
- **Community feedback:** Early PyPI release + GitHub presence to validate whether the "aha moment" resonates broadly or is niche

## Developer Tool Specific Requirements

### Technical Architecture Considerations

**Python version support:**
- Minimum: CPython 3.9+
- `sys.settrace` API is stable across all target versions
- No use of version-gated language features beyond 3.9 baseline

**Packaging:**
- Distributed via PyPI as `coverage-stats`
- `pyproject.toml`-based packaging (PEP 517/518)
- Declares pytest as a dependency; no other runtime dependencies
- Entry point registered as a pytest plugin via `entry_points` (no manual plugin registration required by users)

### Installation Methods

```bash
pip install coverage-stats
# or
uv add --dev coverage-stats
```

Configuration in `pyproject.toml`:
```toml
[tool.coverage-stats]
source = ["src/"]           # directories to measure
output_dir = "coverage-stats-report"  # default output directory
```

Run:
```bash
pytest --coverage-stats
```

### API Surface

**`@covers` decorator:**

```python
from coverage_stats import covers

# Accept a Python object
@covers(MyClass.my_method)
def test_something(): ...

# Accept a dotted string reference
@covers('mymodule.MyClass.my_method')
def test_something(): ...

# Accept a list of either
@covers(MyClass.my_method, 'other.Module.function')
def test_something(): ...

# Apply to a test class (all methods inherit the annotation)
@covers(MyClass)
class TestMyClass: ...
```

**pytest plugin flags:**

| Flag | Description |
|---|---|
| `--coverage-stats` | Enable coverage-stats for this run |
| `--coverage-stats-output=DIR` | Override output directory (default: `coverage-stats-report/`) |
| `--coverage-stats-format=html,json,csv` | Select output formats (default: all) |

**Output files:**

| File | Format | Description |
|---|---|---|
| `coverage-stats-report/index.html` | HTML | Summary index with folder-collapsible layout |
| `coverage-stats-report/<module>.html` | HTML | Per-file line-level view |
| `coverage-stats-report/coverage-stats.json` | JSON | Complete metric set |
| `coverage-stats-report/coverage-stats.csv` | CSV | Flat metric set for CI/dashboarding |

### Code Examples

**Minimal working example (README lead):**

```python
# app/pricing.py
class PricingCalculator:
    def calculate_discount(self, price, user_tier):
        if user_tier == 'premium':
            return price * 0.8
        return price

# tests/test_pricing.py
from coverage_stats import covers
from app.pricing import PricingCalculator

@covers(PricingCalculator.calculate_discount)
def test_premium_discount():
    calc = PricingCalculator()
    assert calc.calculate_discount(100, 'premium') == 80.0
```

Run `pytest --coverage-stats`. The report shows:
- `app/pricing.py`: 100% deliberate coverage, 0% incidental coverage, deliberate assert density: 0.5

Without `@covers`, the same test produces: 100% incidental coverage, 0% deliberate coverage.

**Error path example:**

```python
@covers('app.pricing.OldCalculator.calculate')  # class no longer exists
def test_old_path():
    ...
# → CoverageStatsResolutionError: Cannot resolve 'app.pricing.OldCalculator'
#   — 'OldCalculator' not found in module 'app.pricing'
```

### Implementation Considerations

- **`sys.settrace` profiler** must correctly handle nested calls, generators, async functions, and context managers — missed attributions corrupt all metrics
- **Assert counting** must include assertions in test helper functions called transitively by `@covers`-annotated tests
- **Stateless runs** — each invocation writes fresh output; no database or persistent state between runs
- **Debugger/profiler coexistence** — developers may run pdb, pydevd, or other `sys.settrace`-based tools concurrently; the tool must detect existing trace functions and either chain correctly, emit a warning, or document the limitation clearly

### Risk Mitigation Strategy

**Technical Risks:**
- *`sys.settrace` correctness across edge cases (generators, async, nested calls)* — mitigate by comprehensive test suite on the profiler itself; the tool is tested with `@covers` on its own code
- *Performance overhead making the tool impractical* — mitigate by benchmarking early; document overhead; provide `--no-coverage-stats` escape hatch
- *Debugger/profiler conflicts* — `sys.settrace` can conflict with debuggers (pdb, pydevd) and other profilers that also use `sys.settrace`; the tool must detect existing trace functions and either chain them correctly, emit a warning, or document the limitation clearly

**Market Risks:**
- *Concept too novel — developers don't grasp the value without trying it* — mitigate with README-first "coverage illusion" example that shows the before/after contrast in 10 lines of code
- *Annotation cost perceived as too high* — mitigate by emphasising incremental adoption; even 5% of tests annotated produces actionable insight

**Resource Risks:**
- *Solo maintainer bandwidth* — MVP is intentionally minimal; no server, no persistent state, no dependencies beyond pytest; maintenance burden is low
- *Scope creep before launch* — Phase 2/3 features are explicitly deferred; the MVP boundary is well-defined

## Functional Requirements

### Test Annotation

- **FR1:** A test author can declare that a test deliberately covers one or more code units using a decorator applied to a test function or method
- **FR2:** A test author can specify a code unit by providing a live Python object (function, method, class, or module)
- **FR3:** A test author can specify a code unit by providing a dotted string reference (e.g., `'mymodule.MyClass.my_method'`)
- **FR4:** A test author can specify multiple code units in a single decorator application using a list of objects, strings, or a mix of both
- **FR5:** A test author can apply a coverage declaration to a test class so all test methods in the class inherit it
- **FR6:** The system resolves coverage references lazily, immediately before the annotated test executes
- **FR7:** The system raises an exception and marks the test as failed when a coverage reference cannot be resolved at runtime
- **FR8:** The test suite continues executing after a resolution failure — only the individual test with the bad reference is failed

### Execution Profiling

- **FR9:** The system tracks every source line executed in the measured codebase during each test
- **FR10:** The system attributes each line execution to the correct deliberate or incidental bucket based on the active test's coverage declarations
- **FR11:** The system counts the total number of assertions fired by each test during its execution
- **FR12:** The system correctly profiles line executions through nested function calls, generators, async functions, and context managers
- **FR13:** The system operates with no dependency on coverage.py at any layer

### Metric Computation

- **FR14:** The system computes per-line incidental execution count (executions by tests not claiming coverage of that line)
- **FR15:** The system computes per-line deliberate execution count (executions by tests claiming coverage of that line)
- **FR16:** The system computes per-file incidental coverage percentage (lines with ≥1 incidental execution ÷ total executable lines)
- **FR17:** The system computes per-file deliberate coverage percentage (lines with ≥1 deliberate execution ÷ total executable lines)
- **FR18:** The system computes per-file incidental assert density (total assertions from incidental tests ÷ total lines in file)
- **FR19:** The system computes per-file deliberate assert density (total assertions from deliberate tests ÷ total lines in file)
- **FR20:** The system aggregates all file-level metrics to folder-level metrics

### HTML Reporting

- **FR21:** The system generates an HTML index page listing all measured files with their four metrics (incidental coverage %, deliberate coverage %, incidental assert density, deliberate assert density)
- **FR22:** A report viewer can expand and collapse file listings grouped by folder in the HTML index
- **FR23:** The HTML index displays aggregated metrics for each folder row
- **FR24:** A report viewer can navigate from the HTML index to a per-file detail page
- **FR25:** The per-file HTML page displays each source line annotated with its incidental execution count, deliberate execution count, and assert contributions
- **FR26:** The HTML report is a self-contained static file bundle openable in any browser without a server or internet connection

### Machine-Readable Export

- **FR27:** The system generates a JSON file containing all collected metrics for all measured files and folders
- **FR28:** The system generates a CSV file containing all collected metrics in a flat tabular format suitable for CI integration and dashboarding
- **FR29:** JSON and CSV exports are written to the configured output directory alongside HTML report files

### pytest Integration

- **FR30:** A developer can enable coverage-stats for a test run using a pytest command-line flag
- **FR31:** A developer can configure the source directories to measure via a project configuration file (`pyproject.toml`)
- **FR32:** A developer can configure the output directory for generated report artifacts via project configuration
- **FR33:** A developer can select which output formats to generate (HTML, JSON, CSV) via configuration or command-line flag
- **FR34:** The pytest plugin self-registers automatically via entry points — no manual plugin registration required

### Error Handling & Diagnostics

- **FR35:** The system emits a clear, actionable error message when a coverage reference cannot be resolved, identifying the unresolvable reference, its location in the test file, and a suggested diagnosis
- **FR36:** The system detects when another `sys.settrace`-based tool (debugger, other profiler) is active and either chains with it correctly, emits a warning to the user, or documents the known limitation prominently

## Non-Functional Requirements

### Performance

- **NFR1:** The `sys.settrace` profiler overhead must not increase test suite wall-clock time by more than 2× on a typical Python project (≤1000 tests, ≤50k lines of measured source)
- **NFR2:** Report generation (HTML, JSON, CSV) must complete within 30 seconds for a typical project after test suite execution finishes
- **NFR3:** Memory usage during profiling must not cause out-of-memory failures on machines with ≥4GB RAM for typical project sizes

### Correctness

- **NFR4:** Every source line executed during a test run must be recorded — zero missed line attributions on supported Python constructs (standard functions, methods, generators, async functions, context managers)
- **NFR5:** The deliberate/incidental attribution must be deterministic — the same test suite run under the same conditions must always produce identical metric outputs
- **NFR6:** Assert counts attributed to a line must equal the actual number of `assert` statements executed by the covering test — no over-counting or under-counting
- **NFR7:** Per-file and per-folder aggregated metrics must be mathematically consistent with per-line data (no rounding errors that produce folder totals inconsistent with file totals)

### Compatibility

- **NFR8:** The tool must support CPython 3.9, 3.10, 3.11, 3.12, and 3.13
- **NFR9:** The tool must support pytest 7.x and 8.x
- **NFR10:** The tool must not require any runtime dependency other than pytest — no third-party packages beyond the standard library and pytest itself
- **NFR11:** The generated HTML report must render correctly in current versions of Chrome, Firefox, and Safari without JavaScript errors

### Scalability

- **NFR12:** The tool must handle projects with up to 500 source files and 10,000 tests without crashing or producing corrupt output
- **NFR13:** Memory usage must scale sub-linearly with project size — per-line data structures must not grow unboundedly for large test suites
